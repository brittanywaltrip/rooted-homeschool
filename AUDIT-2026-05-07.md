# Rooted App Audit — May 7, 2026

## Summary

Audit of the staging branch covering dashboard, Plan, Schedule Builder, memories,
yearbook, garden, settings, and shared scheduler/layout code. The Schedule Builder
swap-in is wired correctly at every entry point, but the new sticky "Save / Cancel"
bar collides with the mobile bottom nav and is unreachable on phones, which is the
single highest-risk regression for tonight. Memory save paths bypass the regression
guard in two ways (no direct refreshTodayStory call, plus a busy-guard race against
the 15s polling loop) so Today's Story can stay stale after a save. Free-vs-paid
gating drifts in three places: 30-day memory limit is not enforced, the 50-photo
cap has no upload-time block, and the Plan-page Progress Report download is
ungated. Several smaller UX, copy, and design-system breaks rounded out the list.

## Critical Issues (fix before next user session)

### 1. Schedule Builder bottom bar is hidden behind mobile nav
- **File:** `app/dashboard/plan/schedule/page.tsx:1075`
- **Issue:** Sticky bar is `fixed bottom-0 inset-x-0 ... z-30`. The dashboard
  layout's mobile bottom nav is `fixed bottom-0 ... z-40`
  (`app/dashboard/layout.tsx:497`). On mobile the nav sits on top of the
  Schedule Builder's Save / Cancel / Preview buttons, which means a user
  who builds a schedule on their phone literally cannot tap Save. They
  also can't tap Cancel to back out — the only escape is the browser back
  button.
- **Fix:** Bump the Schedule Builder sticky bar to `z-50` (above the mobile
  nav) and add `bottom-[3.75rem]` (the height of the mobile nav) on
  `md:` breakpoint or below; OR hide the mobile bottom nav while
  `/dashboard/plan/schedule` is the active route. Either way verify on
  iOS Safari with safe-area inset.

### 2. Memory save paths skip refreshTodayStory and race the polling guard
- **File:** `app/dashboard/page.tsx`
- **Issue:** The auto-memory regression rule is "every memory save in
  dashboard must call refreshTodayStory() + loadData()". Today only the
  catch-up modal (`:1345-1346`), `confirmExtraLessons` (`:1961-1962`),
  and `undoReschedule` (`:2366-2367`) follow the rule. Every other save
  path calls `loadData()` alone:
  - `saveBook` — `:2608`
  - `saveDrawing` — `:2639`
  - `saveEditSheet` — `:2673`
  - `deleteFromEditSheet` — `:2682`
  - photo capture file handler — `:3518`
  - field trip / project save — `:3963`
  - win / quote save — `:4823`
  Functionally `loadData()` does call `setTodayStory(...)` at `:1127`, but
  it's gated by `loadDataBusy.current` at `:599`. The 15-second polling
  interval at `:1244-1250` calls `loadData()` and holds the gate for the
  duration of the round-trip. A memory save firing during that window
  is silently dropped — Today's Story does not refresh until the user
  manually navigates away and back.
- **Fix:** After every memory save, await `refreshTodayStory()` directly
  (it doesn't go through the busy gate) AND call `loadData()`. Mirror
  the `confirmExtraLessons` pattern: `loadDataBusy.current = false;
  await loadData(); await refreshTodayStory();`. Apply to all 7 save
  paths above.

### 3. Free users can see all memories beyond the 30-day window
- **File:** `app/dashboard/memories/page.tsx:271`, `:466`
- **Issue:** `dateFloor = "2020-01-01"` fetches every memory regardless
  of plan, and the filter at `:466-484` never applies a 30-day cutoff
  for free users. Per the audit context "Free plan ... only last 30
  days visible". Today a free user keeps seeing every memory they ever
  captured.
- **Fix:** When `planType` is null or "free", set `dateFloor` to
  `localDateStr(thirtyDaysAgo)` AND add a fallback filter (a viewer can
  toggle a filter that shows older memories — keep them out for free
  users). Reflections list also needs the same cutoff (`:289-291`).

### 4. 50-photo limit is not enforced at upload time
- **File:** `app/dashboard/page.tsx:251-288` (FAB save), `:3496-3525`
  (capture file flow), `:2569-2611` (saveBook with cover photo),
  `:2612-2642` (saveDrawing), `app/dashboard/layout.tsx:251-288`
- **Issue:** No path checks `totalPhotos < 50` before inserting a new
  `memories` row with `photo_url`. The Today page already shows a
  warning banner at 45+ photos (`:3318-3329`), but every save path will
  cheerfully insert photo #51, #52, etc. There is no UI message when
  the cap is actually hit.
- **Fix:** Before any photo upload, check `planType` and the current
  photo count via the `memories` query. If `!isPro && totalPhotos >=
  50`, surface a blocking modal pointing at `/upgrade` instead of
  inserting the row. Use the photoCount value already in `loadData`.

### 5. Plan-page Progress Report download is not gated for free users
- **File:** `app/dashboard/plan/page.tsx:3153-3159`
- **Issue:** The download button calls `downloadReport` (`:1630`)
  directly — no `canExport` / `is_pro` check. The standalone Reports
  page at `app/dashboard/reports/page.tsx:391` correctly gates with
  `canExport({ is_pro, trial_started_at })` and pops `ExportGateModal`,
  but the Plan-page version skipped this. CLAUDE.md "Progress Report
  (visible to all, DOWNLOAD paid)" says preview is fine, download is
  paid. `isFreeUser` is even computed at `:455` and then never used.
- **Fix:** Wrap the click handler with the same `canExport` check used
  in the Reports page; show `<ExportGateModal />` for free users.
  Either delete or use the dead `isFreeUser` variable.

### 6. Yearbook reader does not cap free users at 4 spreads
- **File:** `app/dashboard/memories/yearbook/read/page.tsx:1011-1018`,
  `:1147-1213` (mobile), `:1269-1315` (desktop)
- **Issue:** Per the audit context "Yearbook preview = first 4 spreads
  only". The reader builds the full `spreads` array and lets free
  users page through every spread — they only see a `PreviewWatermark`
  overlay (`:1148`, `:1291`) and the print/download is gated. There is
  no hard `spreads.slice(0, 4)` or "Upgrade to view more" wall after
  spread 4.
- **Fix:** Compute `accessLevel = getUserAccess(profile)`. If it equals
  `'free'`, slice `spreads` to 4 and append a sealed "Upgrade to read
  the rest" spread that links to `/upgrade`. Watermark is fine to keep
  on all 4; the cap is the missing piece.

## High Priority (fix this week)

### 7. school_days fallback inconsistent on Today page
- **File:** `app/dashboard/page.tsx:731`
- **Issue:** `(profile as ...).school_days ?? []` falls back to `[]`,
  not the documented Mon-Fri default. Downstream uses guard with
  `if (schoolDays.length > 0)` (`:734`, `:772`, `:801`), so an empty
  array silently becomes "every day is a school day" for streak/dot
  computation. CLAUDE.md and `app/lib/scheduler.ts:801`
  (`normalizeSchoolDays`) both treat null/empty as Mon-Fri.
- **Fix:** Use `normalizeSchoolDays` from scheduler.ts at the read
  boundary, or hard-code the Mon-Fri default literal here. Apply
  same to `app/dashboard/plan/page.tsx:620` and
  `app/dashboard/garden/page.tsx:448`.

### 8. Vacation banner copy implies lessons stay; lessons are still shown
- **File:** `app/dashboard/page.tsx:2882-2915`, `:3065-3071`
- **Issue:** Today does render an active-vacation banner ("Enjoy your
  break — school resumes ..."), but the lessons list still renders
  below ("Logging is optional today" at `:3068`). The audit context
  says "Today page correctly suppress lesson checklist items on
  vacation days". Pre-existing lesson rows whose stored `scheduled_date`
  predates the vacation block still appear because the queue projector
  filter only excludes break days for forward projection, not for
  rows already in the table. Risk: mom on vacation sees the same
  checklist she sees every other day, dampening the "real break"
  feeling.
- **Fix:** Either (a) add a top-level early return that hides the
  schedule entirely on active-vacation days, leaving only the banner,
  or (b) explicitly empty `lessons` and `todayActivities` when
  `activeVacation` is set. Brittany already chose the simpler resume
  date approach for the banner — extend that simplicity here.

### 9. Memory save paths share a 15s polling loop with no debounce
- **File:** `app/dashboard/page.tsx:1244-1250`
- **Issue:** `setInterval(() => loadData(), 15000)` fires every 15s
  regardless of activity. Combined with the busy guard (`:599`),
  this is the root cause of the regression race in Critical #2.
  Even without the regression guard issue, every dashboard tab burns
  ~240 DB roundtrips/hour for users who left it open. The eslint
  override on the dependency array (`:1249`) hints this was rushed.
- **Fix:** Replace the polling loop with `visibilitychange` +
  `rooted:memory-saved` event listeners (memories page already does
  this at `:393-403`), and only re-poll when the tab regains focus.
  Closes the race window and drops backend load.

### 13. Schedule Builder cycleCount allows a "0" badge that the user can't tell from off
